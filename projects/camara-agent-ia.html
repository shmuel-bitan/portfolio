CAMARA & Agent IA

Orchestration autonome de services télécom 5G via les APIs CAMARA et un agent IA piloté par MCP. Détection de fraude et QoS dynamique sans intervention humaine. Module RSX218 · CNAM Paris.

// vue d'ensemble

Ce projet explore l'intersection entre les réseaux mobiles programmables et l'intelligence artificielle décisionnelle. L'objectif est de construire un système autonome capable d'analyser en temps réel l'état d'un réseau télécom et de prendre des décisions automatisées — sans intervention humaine — en s'appuyant sur les APIs standardisées du projet CAMARA et un agent IA orchestré via le Model Context Protocol (MCP).

Il s'inscrit dans une tendance de fond de l'industrie télécom : transformer le réseau en plateforme programmable pilotée par logiciel, où des agents intelligents remplacent les règles statiques par un raisonnement contextuel dynamique.

// contexte technique

CAMARA est une initiative conjointe de la Linux Foundation et de la GSMA visant à standardiser l'exposition des capacités réseau 5G à travers des APIs ouvertes et interopérables. Ces APIs permettent à des applications tierces d'interagir directement avec des fonctions réseau critiques : qualité de service à la demande, détection de changement de SIM, localisation et état des terminaux.

MCP (Model Context Protocol) est un protocole d'interface standardisé permettant à un agent IA d'accéder à des outils externes de manière structurée et sécurisée. L'agent ne code pas de logique : il raisonne, sélectionne les outils pertinents, interprète les résultats et formule une décision.

// architecture — 4 couches

COUCHE 1 — RÉSEAU SIMULÉ

APIs CAMARA émulées via un serveur FastAPI exposant 4 endpoints : SIM Swap Detection, Quality on Demand, Device Status, Device Location. Reproduit fidèlement le comportement d'un opérateur réel.

COUCHE 2 — SERVEUR MCP

Un serveur MCP enveloppe les APIs CAMARA et les expose comme outils consommables par l'agent IA. Interface standardisée entre l'intelligence artificielle et les services réseau.

COUCHE 3 — AGENT IA

Un LLM agit comme moteur de décision. Il reçoit un événement déclencheur, sélectionne et appelle les outils MCP nécessaires, corrèle les informations collectées et produit une décision justifiée.

COUCHE 4 — EXÉCUTION

Les décisions de l'agent sont traduites en actions concrètes : blocage d'une transaction, activation d'une session QoS prioritaire, déclenchement d'une vérification supplémentaire.

// cas d'usage 1 — détection de fraude télécom

Lors d'une transaction sensible, l'agent interroge simultanément trois APIs CAMARA pour croiser plusieurs signaux en temps réel :

  • SIM Swap Detection — changement récent de carte SIM détecté ?
  • Device Status — terminal en roaming, type de réseau, état de charge
  • Device Location — cohérence géographique de la demande

En croisant ces trois signaux, l'agent décide d'autoriser, de bloquer ou de soumettre le paiement à une vérification renforcée. Ce raisonnement multi-source remplace avantageusement une règle statique binaire qui n'exploiterait qu'un seul critère.

# Flux décisionnel simplifié de l'agent event: payment_request(user_id, amount) → call sim_swap_detection(user_id) → call device_status(user_id) → call device_location(user_id) if sim_swap_recent AND location_mismatch: decision = "BLOCK" elif sim_swap_recent OR roaming_anomaly: decision = "CHALLENGE" else: decision = "AUTHORIZE"

// cas d'usage 2 — QoS dynamique intelligente

Lorsqu'une application critique est lancée sur un terminal (streaming vidéo temps réel, appel d'urgence), l'agent évalue l'état réseau courant via Device Status et active automatiquement une session QoS prioritaire via l'API Quality on Demand si les conditions l'exigent. L'activation est contextuelle : elle dépend du type de réseau détecté, du profil applicatif et de l'état de charge du terminal.

// rupture de paradigme

Ce projet met en lumière une rupture fondamentale dans la façon de piloter un réseau. Une approche classique basée sur des règles statiques nécessite d'anticiper chaque scénario à l'avance — elle est rigide, non corrélée et difficile à maintenir à l'échelle.

L'approche agentique renverse ce paradigme : l'agent raisonne sur le contexte, corrèle des signaux hétérogènes en temps réel et adapte ses décisions sans redéploiement. Les enjeux de scalabilité, latence décisionnelle, traçabilité des décisions IA et sécurité des interfaces MCP sont également analysés dans le projet.

// ce que j'ai appris

  • Implémentation d'un serveur MCP et exposition d'outils à un agent IA
  • Conception d'un agent IA décisionnel multi-outils
  • Déploiement d'une architecture conteneurisée multi-couches sous Docker
  • Analyse critique des limites et enjeux de l'IA autonome en contexte télécom
